## Context

Приложение строит диаграмму Вороного и раскрашивает ячейки по текущей палитре без пользовательской настройки оттенка. Требуется добавить простой и предсказуемый способ смещать оттенок всех цветов диаграммы через UI, без изменения остальной логики генерации геометрии.

Ограничения:
- Решение должно использовать текущий стек Flutter без новых внешних зависимостей.
- Изменение должно быть локализовано в UI-состоянии и функции, формирующей цвет ячеек.

## Goals / Non-Goals

**Goals:**
- Добавить контрол настройки hue shift для диаграммы Вороного.
- Применять hue shift ко всем ячейкам сразу после изменения значения.
- Валидировать диапазон значения и не допускать некорректного состояния.

**Non-Goals:**
- Добавление сложного редактора палитры (индивидуальный цвет для каждой ячейки).
- Долговременное сохранение настройки между перезапусками приложения.
- Рефакторинг алгоритма построения диаграммы Вороного.

## Decisions

1. Ввести состояние `hueShift` в экране диаграммы как `double` в градусах (0..360).
   - Почему: значение напрямую соответствует модели HSL/HSV и легко маппится на цветовые преобразования.
   - Альтернатива: 0..1 коэффициент. Отклонено, так как менее наглядно в UI.

2. Добавить UI-контрол на основе `Slider` с отображением текущего значения.
   - Почему: быстрый визуальный подбор оттенка и минимальный объем кода.
   - Альтернатива: числовое поле ввода. Отклонено для базовой версии из-за менее удобного UX.

3. Применять смещение оттенка на этапе вычисления цвета ячейки через `HSLColor`.
   - Почему: встроенный API Flutter обеспечивает корректное преобразование оттенка с сохранением насыщенности и светлоты.
   - Альтернатива: ручное преобразование RGB. Отклонено из-за большей сложности и риска ошибок.

## Risks / Trade-offs

- [Риск] Быстрые изменения слайдера могут вызывать частые перерисовки. → [Снижение] Обновлять только нужное состояние и не пересчитывать геометрию, если меняется только цвет.
- [Риск] Неправильная нормализация (например, >360) даст визуальные артефакты. → [Снижение] Жестко ограничить и нормализовать значение в диапазон 0..360.
- [Компромисс] Настройка действует только в рамках сессии. → [Обоснование] Упрощает первый релиз и минимизирует побочные изменения.

## Migration Plan

1. Добавить новое состояние hue shift и UI-контрол в текущий экран.
2. Обновить функцию формирования цвета ячейки с учетом смещения.
3. Проверить поведение в рантайме на типичных и граничных значениях (0, 180, 360).
4. При необходимости откатить изменения удалением нового состояния и контролла (изменение локальное и обратимое).

## Open Questions

- Нужно ли сохранять hue shift между запусками в последующих итерациях (например, через `shared_preferences`)?
- Нужен ли шаг настройки оттенка более мелкий, чем стандартный шаг слайдера?
